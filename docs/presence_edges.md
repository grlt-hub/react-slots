# Presence: edge cases & boundaries — doc source material

Compiled from the 2026-06-07 re-audit (`ef98d63`) plus the slot-in-slot analysis. Status tags: **[doc]** — already in CLAUDE.md boundaries (number given), **[new]** — found by the audit, missing from the docs, **[unpinned]** — derived from the code, not yet pinned by a test.

## When it works (verified cases — each backed by a pin or by the re-audit)

The general rule: presence answers the question **"did the child commit DOM inside the slot region"** — and for box-level chrome slots (toolbars, panels, menus, sidebars) that is exactly the right question.

1. **The canonical case: chrome only when there is real content.** A separator / section header / panel frame shows on `useCount() > 0`, not on "something was inserted": an inserted plugin that renders `null` (feature flag, permissions, empty data) correctly counts as absent. _(pins: null/`<></>`/false/""/[] → false; `<i/>`/"text" → true)_
2. **Empty-state placeholder.** "No active widgets" when N plugins are inserted but all of them rendered `null`. The mirror of case 1.
3. **A badge counting what actually rendered.** `useCount()` — the number of children actually rendering, updated live without re-rendering the others. _(pin: independent subscribers — each host receives its own update)_
4. **Self-toggling children.** A child flips `null ↔ content` from its own state — the observer catches it, the probe wrapper never re-renders, the count follows. _(pin: child toggling … live-updates; the probe wrapper never re-renders)_
5. **Children with `filter`.** Bootstrap-reject → false (the span is absent); first pass → true; freeze (filter turned false) → DOM stays alive, still true; thaw → no remount. Presence composes correctly with the entire filter contract, independently per Root. _(pins: 4 bootstrap/pass/freeze/thaw tests + per-Root independence)_
6. **Multi-Root.** One slot mounted in several places — OR semantics with no double-counting: "renders somewhere". Unmounting one Root does not drop the count; unmounting all → 0. _(pins: 3 multi-Root tests)_
7. **`usePresence()` for per-item indication.** Which of the inserted plugins actually rendered — an array in composition order (order, then insertion order); a stable reference while nothing has flipped → cheap memoized hosts. _(pins: array follows slot order; stable reference)_
8. **Lazy/async plugins.** A `React.lazy`/Suspense child: false during the fallback, true after commit; initial-suspend → 0, after resolve → N. _(verified by the re-audit, concurrent vector)_
9. **Portals with a local anchor.** A modal plugin with a local button plus `createPortal` elsewhere — counts (the anchor is in the region); a portal-only child does not. For region semantics this is the desired behavior: a toolbar does not draw its frame for the sake of an invisible modal. _(pins: portal-only → false; portal + local → true)_
10. **SSR + hydration.** The server emits count=0 / all-false length-matched, hydration stays silent, the `0 → N` settle lands on its own — render the chrome optimistically and no polish is needed. _(pins: SSR length-match + hydration silent)_

**etc.:** enabling/disabling plugins at runtime (insert/clear, 500 cycles — count exact, host re-renders constant); StrictMode (double effects never double-count); `startTransition` over flips (every recorded count/array pair is consistent, no tear observed); feature flags / routing inside children (a special case of 4); independent count-host and presence-host on one slot; "are there plugin toolbar buttons" in several layout spots at once (a special case of 6); deep content updates under a persistent wrapper never touch the host at all (childList-only observer + the Object.is gate — zero phantom re-renders).

## Slot-in-slot

**The base case works.** [new, unpinned] The probe measures `span.childNodes.length > 0` and does not care who produced the DOM. A child rendering `<Inner.Root/>` with no wrapper:

- the inner slot is empty / all of its children render `null` → nothing commits inside the outer span → outer presence is **false**;
- an insert into the inner slot → the inner Root's children land as direct DOM children of the outer span (arrays/fragments create no DOM nodes) → a childList mutation on the span itself → the observer fires → **true**; `clear()` — back to false. Live updates cascade correctly.

**Gotcha #1 — a wrapper (a special case of has-DOM, [doc] the deep-mutation boundary):** `<div><Inner.Root/></div>` — the div is always there → outer presence is always true regardless of what happens inside the inner slot.

**Gotcha #2 — presence-in-presence (THE KEY ONE, [new, unpinned], candidate for boundary #12):** if the inner slot is also `{presence: true}`, its children are wrapped in probe spans, and a probe span is a real `Element`. An inner slot with a single child rendering `null` → the outer span contains an empty `<span style="display:contents">` → `childNodes.length === 1` → **the outer count reads true with zero actual content**. The outer presence degrades from "there is DOM inside" to "the inner slot has at least one insert". Formally consistent with the has-DOM definition ("produced any DOM node"); semantically a false-positive cascade. Doc recommendation: **do not enable presence on both levels of nested slots**; if you did, read the outer count as "inner slot non-empty", not as "inner content exists".

## The remaining edges (summary table)

| `tr`/`td`/`th` (table), `option` (select), `source`/`track`, SVG internals        | the span gets foster-parented out by the parser — rendering itself breaks; presence is for box-level chrome slots only                                                                                                                                                                                                                                                                                                                                  | [doc] #3                              |
| Portal-only child (`createPortal` elsewhere)                                      | counts as **not** rendering (region semantics); portal + local sibling content → true                                                                                                                                                                                                                                                                                                                                                                   | [doc] #1                              |
| `display:none` / `visibility:hidden` / zero size / offscreen                      | **counts** as rendering: has-DOM ≠ visible; visual visibility is a separate feature, out of scope                                                                                                                                                                                                                                                                                                                                                       | [doc] #2                              |
| A persistent wrapper in the child (`<div>{cond ? x : null}</div>`)                | always true; deep flips change neither the value nor host renders (childList-only observer + the Object.is gate)                                                                                                                                                                                                                                                                                                                                        | [doc] (deep-mutation pin)             |
| Text nodes and comments                                                           | `" "` → true (a text node is DOM); `null`/`""`/`[]`/`<></>` → false                                                                                                                                                                                                                                                                                                                                                                                     | [doc] #10                             |
| Suspense hydration comment markers (`<!--$-->`)                                   | a dehydrated Suspense boundary inside a child keeps comment nodes in the DOM → the probe sees childNodes → true even with a null fallback. A concrete present-day instance of the "accepted edge" in #10 — worth naming explicitly                                                                                                                                                                                                                      | [new, unpinned]                       |
| **Suspense re-suspension** (a sibling in the same boundary suspends after commit) | React hides the content with `display:none` AND tears down the probes' layout effects (`observer.disconnect` + `remove`) → count/array transiently go 0/all-false while the live (hidden) DOM persists; recovers exactly to N on reveal (verified over 3 cycles). The same is expected from `<Activity mode="hidden">`. Contradicts a literal reading of #2 ("display:none counts") — only user-applied display:none counts, not React-Offscreen hiding | [new] (re-audit, I3; repro available) |
| SSR / first paint                                                                 | server and client first render: count=0, all-false length-matched; the `0 → N` settle lands via microtask+effect. A UI hard-gated on `count === 0` will flicker once — render optimistically                                                                                                                                                                                                                                                            | [doc] #5                              |
| An insert between `renderToString` and `hydrateRoot`                              | a genuine recoverable hydration mismatch (server length-2 vs client length-3), full recovery. Composition must be stable across the SSR→hydrate window — the usual SSR contract (plugins register before render)                                                                                                                                                                                                                                        | [new] (re-audit, I4; repro available) |
| Host CSS: `.slot > *`, `+`, `:nth-child`                                          | matches the probe spans, not the content; breaks styling (not presence values). Prefer `gap` over adjacent-sibling margins                                                                                                                                                                                                                                                                                                                              | [doc] #4                              |
| Multi-Root + the teardown window                                                  | OR across mounted Roots; on clear/unmount, `reports` transiently holds stale entries, but the projection over `store.get()` never yields a wrong value                                                                                                                                                                                                                                                                                                  | [doc] #7                              |
| Mid-list insert                                                                   | `usePresence()` is a snapshot in composition render order; indices shift                                                                                                                                                                                                                                                                                                                                                                                | [doc] #8                              |
